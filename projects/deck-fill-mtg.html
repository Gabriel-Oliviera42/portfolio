Deck Fill MTG

Uma ferramenta completa criada para jogadores de Magic: The Gathering. O objetivo da aplicação é permitir que os usuários busquem cartas, montem seus decks personalizados e gerem arquivos otimizados para a impressão de proxies para jogos casuais.

O que exatamente é isso?

Se você não é do mundo dos Card Games, pode estar se perguntando o que é esse projeto. Magic: The Gathering (MTG) é um dos jogos de cartas mais complexos e caros do mundo. Muitas vezes, os jogadores querem testar um baralho (deck) novo antes de gastar dinheiro comprando as cartas originais. É aí que entram as Proxies.

Uma "Proxy" nada mais é do que uma impressão não-oficial de uma carta, usada exclusivamente para testes e jogos casuais com amigos. O problema é que procurar imagens em boa qualidade, ajustar o tamanho correto para impressão e organizar tudo em uma folha A4 dá um trabalho absurdo.

O Deck Fill nasceu para resolver exatamente essa dor. Ele é um sistema onde o usuário pesquisa as cartas, adiciona ao seu deck virtual e, com um clique, o site organiza e gera tudo pronto para a impressora. Rápido, prático e sem dor de cabeça.

O Desafio Técnico (E como o projeto cresceu)

  1. Construção do Front-end. A interface precisava ser intuitiva. Criei um sistema de busca em tempo real e um gerenciador de estado para que o usuário pudesse ver seu deck sendo montado dinamicamente antes de enviá-lo para impressão.
  2. Desenvolvimento da API (Back-end). Para não depender de APIs lentas de terceiros, construí meu próprio back-end em Python utilizando o framework FastAPI. O servidor precisava processar buscas rápidas e entregar as imagens corretas.
  3. Lidando com um Volume Massivo de Dados. Foi aqui que o projeto tomou proporções gigantescas. Inicialmente focado em MTG, decidi expandir o banco de dados (SQLite) para suportar outros jogos como Yu-Gi-Oh!, Pokémon e Lorcana. O resultado? Mais de 600 MB só de bancos de dados locais!
  4. Pausa Estratégica. Gerenciar infraestrutura gratuita para quase 1 GB de banco de dados se tornou inviável. Como desenvolvedor, precisei tomar a decisão madura de congelar a escalabilidade do back-end, garantir que a versão atual ficasse estável e documentar o aprendizado formidável que tive ao lidar com um projeto desse tamanho.

O que esse projeto me ensinou

Gerenciamento de Escopo

Aprendi na prática o que significa "Scope Creep". Querer adicionar suporte para 7 jogos diferentes me mostrou como o excesso de funcionalidades pode travar o deploy de um software. Entregue o MVP primeiro!

APIs e Back-end em Python

Tive uma imersão profunda na criação de servidores com FastAPI, entendendo rotas, requisições HTTP, CORS e comunicação direta do servidor com o banco de dados.

Limitações de Infraestrutura

Descobri da forma mais técnica possível como funcionam as restrições de plataformas em nuvem (como Render, Vercel e Fly.io) e limites de versionamento no GitHub para arquivos gigantes.

Bancos de Dados Relacionais

Trabalhar com arquivos SQLite pesados me forçou a entender como estruturar os dados para que o site conseguisse realizar as buscas pelas cartas de forma eficiente.

A ilusão do "É só consumir a API"

Quando decidi expandir o site para 7 jogos diferentes, eu fui pela emoção. Achei que seria apenas plugar as APIs de cada Card Game e pronto. A realidade cobrou caro. Como eu não estudei a documentação de cada uma previamente, me deparei com problemas bizarros durante o desenvolvimento: uma API me retornava imagens com marcas d'água gigantes inutilizando a proxy, enquanto outra tinha um sistema de busca tão confuso e engessado que eu precisava fazer malabarismos no código só para o usuário conseguir achar uma carta simples.

A falta de um cuidado anterior para tratar esses dados transformou meu back-end em uma verdadeira colcha de retalhos na tentativa de contornar as limitações de cada provedor, claro que não é o unico problema e com certeza não percebi tudos os meus erros, mas só de olhar para tras e ver que eu sei oque eu deveria fazer já me da vontade de fazer um projeto desse tamanho mesmo sabendo que vou cometer outros erros para aprender com eles

O que eu farei no futuro (Visão 2.0)

Olhando para esse projeto hoje, eu vejo perfeitamente onde errei pela pressa de ver tudo funcionando na minha tela. Mas é exatamente por isso que considero esse site um marco no meu aprendizado.

No futuro, pretendo reescrever este projeto com o carinho e a maturidade técnica que ele exige. A minha abordagem para a versão 2.0 será completamente diferente:

Estudo profundo de cada API: Antes de escrever qualquer linha de código, vou ler as documentações de cabo a rabo. Preciso entender os limites de requisição (rate limits), o padrão de resposta e como higienizar os dados para garantir que nenhuma imagem com marca d'água ou texto quebrado chegue ao usuário.

Arquitetura antes do Código:Vou desenhar a modelagem dos dados primeiro. Em vez de baixar o mundo inteiro e gerar 600 MB de banco de dados local para contornar buscas ruins, vou construir um sistema de consumo inteligente e cache eficiente.

Um jogo por vez: Aprendi que a complexidade escala rápido demais. Vou focar em refazer a infraestrutura para que um único jogo (o MTG) rode de forma impecável, profissional e escalável, antes de sequer pensar em adicionar o segundo.
